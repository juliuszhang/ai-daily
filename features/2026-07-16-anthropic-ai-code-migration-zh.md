*一份使用 AI Agent 执行大型代码迁移的分步指南——其中包括 Bun 从 Zig 到 Rust、规模达百万行代码的迁移。*

代码迁移，也就是把生产代码库移植到另一种语言的项目，直到最近都还是需要耗费数年的大型工程。

过去一个月里，Anthropic 的个别开发者使用 Claude Fable 5、Claude Opus 4.8 和[动态工作流](https://claude.com/blog/introducing-dynamic-workflows-in-claude-code)，迁移了 10 个代码包，每个包包含数万到数十万行代码。本文将介绍其中两个案例，以及从这些项目中总结出的最佳实践。

Bun 联合创始人、Anthropic 技术人员 Jarred Sumner 使用 Claude Code [将 Bun 从 Zig 迁移到 Rust](https://bun.com/blog/bun-in-rust)。不到两周便产出了 100 万行代码；合并前，Bun 现有测试套件在 CI 中的通过率达到 100%。合并后发现了 19 个回归问题，目前均已修复。Rust 移植版本已于 6 月随 Claude Code 一同发布。

Anthropic Labs 联合负责人 Mike Krieger 在一个周末内将一个 Python 代码库迁移成了 16.5 万行 TypeScript。整个过程使用了数百个 Agent、8 个阶段门、3 轮对抗式审查，并在最后执行一致性检查，将每条命令的输出与原 Python 版本逐一进行差异比较。

Claude Code 的新能力改变了这些长期搁置项目的成本收益计算。下面这套六步流程，正是我们从上述迁移项目中总结并投入使用的方法。

核心洞见是：**不要修代码本身，而要修复生成这些代码的流程（循环）。**

## 为什么迁移语言，以及什么时候迁移

在直接讨论“怎么做”之前，值得先讨论“何时做”和“为什么做”，因为人们对这类项目的基本判断已经发生了变化。

团队启动迁移，通常是因为项目从最初构建到当前阶段所处的技术环境发生了变化：原先已知的取舍开始成为限制，更好的方法已经出现，或者原有生态正在萎缩。

例如，Jarred 最初选择 Zig，是因为它既能提供 C 级性能，又具有极致的简洁性，非常适合一位独立创始人“在大模型出现之前，窝在奥克兰一间狭小公寓里，用一年时间写出 Bun”。这种简洁性也伴随着一些已知取舍，他在[这篇文章](https://bun.com/blog/bun-in-rust#just-be-really-smart-and-don-t-make-mistakes)中对此进行了说明。

时间来到 2026 年，Bun CLI 的月下载量已经超过 1,000 万次，并在 Claude Code 中得到广泛使用。

就在上个季度，这些取舍仍不足以让团队冻结产品路线图，并为一个持续数个季度的项目投入资源。语言迁移可以带来更小、更快、更安全的系统，但没有人愿意承担它过去那样高昂的代价。

软件工程师还必须面对这类超级项目所固有的职业风险。团队可能需要同时维护两套代码数个季度甚至数年；如果最后只实现了 90% 的行为一致性，结果往往比迁移前更加棘手。

现在，最坏的情况只是删掉这个分支，再试一次。

当然，迁移仍需具备合理的商业理由。百万行级迁移虽然不再意味着一项持续四年、耗费 300 万至 400 万美元工程资源的项目，但执行成本仍可能达到数万、数十万美元，甚至更高。例如，Bun 的迁移消耗了 59 亿个未缓存输入 Token 和 6.9 亿个输出 Token；按 API 价格计算，成本约为 16.5 万美元。Mike 的移植项目主体则使用了 2,700 万个 Token。

*Jarred 的百万行代码 PR。*

**不过，迁移的理由已不再需要上升到“生死存亡”的程度。** 如果变更日志里一整年都在修复内存错误，或者系统长期存在某个顽固瓶颈，现在就可能足以证明迁移值得进行。

编译环节正是 Mike 启动项目的原因。他所在团队的内部工具以单个二进制文件交付给用户。使用 Python 工具链时，每个平台生成二进制文件大约需要 8 分钟；每次发布在整个构建矩阵上总计要等待约 30 分钟。迁移完成后，同样的编译只需要约 2 秒，二进制文件启动速度提高了 6 倍，团队也得以淘汰一套独立的部署流水线。

## 为什么 AI 改变了代码迁移的成本收益计算

Claude Fable 5 是我们目前能力最强、已全面可用的模型。Fable 和 Opus 4.8 尤其擅长通过子 Agent 委派、指挥和验证并行工作流，同时为既定目标寻找多条实现路径。

大型代码迁移特别适合使用这些先进模型，原因如下：

- **工作可以并行。** 任务能够拆成数千个彼此独立的单元，例如文件和 crate；多个 Agent 可以同时工作，而不必互相等待。
- **上下文明确且完整。** 旧代码本身就是模型极好的规格说明，也能作为核心参考，帮助编写供迁移 Agent 遵循的指南。
- **系统自带裁判。** 许多大型代码库已经有测试套件，Agent 可以用它验证工作成果。当验证标准客观时，Agent 表现最好，因为模型可以连续数天围绕一个真实基准反复迭代，而不需要人类不断判断质量。
- **任务队列会自动生成。** 编译或测试一旦失败，就自然成为 Agent 的下一项修复任务。
- **迁移要求一致性并妥善处理边缘情况。** 整个流程要让偏差无处隐藏：审查者必须为每项发现引用对应规则，这样，违反规则就会成为队列中的明确任务，而不会悄悄演化成实现偏差。当 Agent 遇到边缘情况时，修复方案会成为一条新规则，供之后所有 Agent 遵循。

下文会看到，Mike 和 Jarred 都在迁移流程的关键步骤中使用了 Fable，尤其采用了一种“顾问模式”：组合不同等级的模型，以优化 Token 消耗。

## 大规模代码迁移的六个步骤

下面的流程已经过泛化，可应用于不同语言和场景。更多细节可参阅 [Jarred 的博客](https://bun.com/blog/bun-in-rust)和 [Migration Starter Kit](https://github.com/anthropics/code-migration-kit-with-claude-code)。需要说明的是，Starter Kit 是对本文流程的通用化模板，并不是上述两个具体移植项目当时实际使用的完整系统。

### 前置条件：建立可靠的裁判机制

在开始迁移项目之前，必须先建立一个可靠的裁判机制；否则，你既没有明确的退出条件，也无法衡量是否成功。

这个裁判必须能以同样的标准评估原始代码和目标代码。用原语言编写的测试套件往往依赖内部函数，而这些内部函数可能不会出现在目标代码中。

建立裁判机制可分为三步：

- **为现有测试分类。** 使用 Claude 识别哪些测试能够表达为外部调用，哪些测试依赖无法移植的内部实现。
- **将测试重写为可移植形式。** 把面向外部行为的测试转成一组可同时对原版本和移植版本运行的断言。再让对抗式 Agent 验证重写后的测试没有削弱断言强度。
- **验证裁判本身。** 先在原代码上运行，确认测试能够通过；然后在故意破坏过的代码上运行，确认它确实会失败——发现不了故障的裁判，不能算裁判。

Jarred 已经拥有一套使用第三种语言 TypeScript 编写的大型测试套件，但大多数项目并没有这样的条件。在 Python 到 TypeScript 的移植中，Mike 为 7 个真实场景建立了一致性验证框架，并把任何行为变化都视为必须修复的错误。

在逐步介绍每个阶段之前，可以先从整体流程理解两种方法。Jarred 的方法在每个阶段都设置审查和阶段门。Mike 使用了相似的总体结构与循环工作流，但他会完整地从头到尾执行一次迁移，再根据结果修改规则和工作流，然后重新运行；前两次输出都被丢弃，直到第三次运行才保留结果。

### 第 1 步：建立规则手册、依赖图和差距清单

这一阶段要为迁移打好基础，包括：列出哪些位置需要重构而不只是翻译；建立代码转换规则手册；绘制依赖图，用来安排迁移实施各工作流的先后顺序。

顺序非常重要：必须先建立规则手册，再编制差距清单。差距清单描述的是规则手册默认规则无法覆盖的内容，二者之后还需一起接受联合审计。

#### 规则手册

[规则手册](https://github.com/anthropics/code-migration-kit-with-claude-code/blob/main/templates/RULEBOOK.md)的具体形式取决于项目一开始必须作出的关键架构决策，其中最重要的是：新代码是否继续沿用原有结构，还是彻底重新设计。

如果沿用原结构——如 Jarred 的项目——规则手册主要是一系列查找表，用于映射两种语言的类型和惯用写法，并把较难迁移的组件指向差距清单。如果选择重新设计——如 Mike 的项目——规则手册更接近一份设计文档。

Jarred 通过与 Claude 对话建立规则手册，为每个存在歧义的领域制定策略。他还根据自己的经验设计了 8 个子 Agent，分别针对 8 类常见失败模式进行审查。

#### 依赖图

要把迁移有效拆分成多个并行工作流，就必须理解文件之间的依赖关系，从而知道哪些文件应当优先迁移、哪些文件应当放进同一批次。

有些语言和代码库具备明确的清单文件，依赖分析相对容易；但对于遗留代码库，以及 C、C++、Python 等许多流行语言，依赖关系通常需要另外发现和绘制。

Claude Code 可以部署 Agent，创建并运行一个确定性脚本来生成依赖图。Migration Kit 中的[依赖图提示词](https://github.com/anthropics/code-migration-kit-with-claude-code/blob/main/prompts/01-dependency-map.md)通过工作流建立了一个“审查—修复”循环。再次说明：Starter Kit 是本文方法的通用模板，并非两个具体移植项目实际使用的原始系统。

#### 差距清单与质疑型审查 Agent

新语言有不同于旧语言的约束，迁移必须满足这些约束。从 Zig 到 Rust 的关键差异之一是手动内存管理；C 和 C++ 也有类似问题。例如：

**Zig：**

```zig
fn readConfig(allocator: std.mem.Allocator) ![]u8 {
    const buf = try allocator.alloc(u8, 1024);
    // ...填充 buf...
    return buf; // 调用方必须释放它——但只有这条注释说明了这一点
}

// 即使调用方忘记执行 `defer allocator.free(buf)`，代码仍能编译；
// 内存泄漏只有在运行时才会显现。
```

**Rust：**

```rust
fn read_config() -> Vec<u8> {
    let buf = vec![0u8; 1024];
    // ...填充 buf...
    buf // 所有权转移给调用方；内存将自动释放
}

// 所有权转移后继续使用？重复释放？两者都无法通过编译。
// 忘记释放？根本没有需要记住的 free 调用——drop 会自动执行。
```

从 Python 迁移到 TypeScript 时，主要差距是接口和契约。Python 不要求预先声明函数接受什么形状的对象、返回什么内容，但 TypeScript 要求这样做。例如：

**Python：**

```python
def register(handler):
    handler.setup()
    return handler.run({"retries": 3})

# 任何具有 .setup() 和 .run() 的对象都可以在这里工作。
# 实际上传入了哪些对象？需要阅读整个代码库才能找出来。
```

**TypeScript：**

```typescript
interface RunResult {
  ok: boolean;
}

interface Handler {
  setup(): void;
  run(opts: { retries: number }): Promise<RunResult>;
}

function register(handler: Handler): Promise<RunResult> {
  handler.setup();
  return handler.run({ retries: 3 });
}

// 必须先明确写出契约，这段代码才能通过编译。
```

Jarred 和 Mike 都建立了差距清单文件，用于记录这些隐性知识。Jarred 选择像本文流程一样预先盘点差距；Mike 则选择先翻译，再通过事后审计形成差距清单。实际项目中，两种方式可能都需要使用。

这里有一个用于生成差距清单的 [Claude Code 示例提示词](https://github.com/anthropics/code-migration-kit-with-claude-code/blob/main/prompts/02-gap-inventory.md)。

### 第 2 步：对规则进行压力测试

这一步会执行一次小规模迁移，作为正式迁移前的“试航”。

Jarred 使用一个 Agent 按规则手册翻译 3 个文件，另一个 Agent 以“资深 Rust 工程师”的方式翻译 3 个文件，再由第三个 Agent 根据两份结果的差异生成新的迁移规则。在这个阶段，他发现了两个关键问题；如果直接把任务扩散到全部 1,448 个文件，这两个问题会制造大量错误。

提示词可以参考 [Migration Kit 中的示例](https://github.com/anthropics/code-migration-kit-with-claude-code/blob/main/prompts/03-stress-test.md)。

这种压力测试只适用于**保持原结构的迁移**，因为同一文件的两份翻译结果需要能够逐行比较。如果规则手册描述的是一次重新设计——如 Mike 的项目——对应做法就是让对抗式审查 Agent 直接攻击设计文档，再通过一次可丢弃的端到端运行验证设计。

无论使用哪种方式，都要丢弃这一阶段产生的迁移文件。这里的目标是完善规则，而不是积累迁移进度。

### 第 3 步：翻译全部代码

接下来的步骤都使用相同的多 Agent 循环架构：**实施、审查、修复。**

可以把实施工作交给较小的模型，把审查工作留给较大的模型。例如，Mike 在主要迁移阶段并行启动了 12 个子 Agent，并使用 Claude Sonnet 执行实施工作。

工作队列应当完全机械化。批处理脚本通过检查磁盘上是否存在对应的已翻译文件来判断任务是否完成，再把待处理文件分成多个批次交给实施 Agent。由于每次都从磁盘重新构建队列，整个迁移天然具备断点续跑能力。

这一阶段的 Agent 有时会过于谨慎，不愿一次完成足够多的工作。解决方法可以很直接：在提示词中明确、强烈地要求它继续推进，并说明编译器会在下一阶段捕获错误。

翻译 Agent 无法确信能够正确处理的内容，统一标记为：

```text
// TODO(port): <原因>
```

这些项目将在第 4 步处理。从这里开始，待办队列会自行生成：编译器列出错误，冒烟测试发现崩溃，测试套件报告失败。

两名对抗式审查 Agent 在彼此隔离的上下文中评估实施 Agent 的工作；如果两名审查者意见不一致，再交给第三个 Agent 裁决。当审查者在多个文件中反复发现同一错误时，不应逐文件修补，而应在规则手册中增加一句规则，然后重新生成受影响的批次。

规则手册会在这个阶段持续增长；代码绝不应脱离规则手册被手工打补丁。

这一阶段还有一个重要设计决定：编译器应放在循环中的什么位置。Mike 把 TypeScript 编译器放进每次循环，因为检查一个单元只需几秒。Jarred 则完全禁止在这一循环中调用编译器，把编译推迟到下一步，因为 Cargo 编译需要数分钟。

走到这里，大部分重活已经完成，[提示词也开始变得更短](https://github.com/anthropics/code-migration-kit-with-claude-code/blob/main/prompts/04-translation-kickoff.md)。

### 第 4、5、6 步：编译、运行并匹配行为

这三个步骤使用相同的循环架构，而且对人类判断的需求会逐步下降，因此放在一起介绍。

#### 第 4 步：编译

根据迁移语言和项目规模的不同，第 4 步有时会直接融入第 3 步。

如果全量编译规模很大或过程复杂，Agent 可能完全不负责触发编译。Jarred 使用一个编排脚本，在整个工作区只调用一次编译器。随后，“修复 Agent”并行处理错误列表，同时接受对抗式审查；然后重新构建，不断重复这一过程。

审查错误列表有助于发现需要系统性调整的问题。例如，Jarred 修复 Zig 惰性编译所容忍的循环导入后，遇到了数千个 Rust 模块错误。他没有逐个处理，而是在循环中加入分类逻辑，判断每项依赖应该删除、移动，还是通过重新划分边界来解决。

#### 第 5 步：运行

和编译错误列表一样，第 5 步也有一个机械化的事实来源：冒烟测试中出现的崩溃。对循环的改进仍然是对问题进行分类——这次按照根因对故障进行分组，再交给对抗式子 Agent 审查。

#### 第 6 步：匹配行为

最后一步是在两个代码库之间比较程序行为。

此时文件已经完成翻译、编译和冒烟测试。接下来把测试分片，针对这些分片运行前置阶段准备好的测试套件。每项失败由“修复 Agent”同时对照两个代码库和失败测试进行分析，对抗式审查 Agent 再检查修复结果。

这个循环的下一阶段引入了一个[构建守护进程](https://github.com/anthropics/code-migration-kit-with-claude-code/blob/main/scripts/build_daemon.sh)，它是唯一允许重新构建二进制文件的进程。修复 Agent 只负责写补丁；守护进程把补丁合并成批次，只重新构建一次，然后重新运行受影响的测试，并把结果反馈给 Agent。这样可以把成本最高的操作串行化，避免多个 Agent 各自重复触发构建。

如果同一个失败在大量测试中反复出现，修复就要向上游移动：修改导致错误的规则，只重新生成受该规则影响的文件。

Mike 的做法在这里尤其有参考价值，因为很多开发者并没有完整、可移植的测试套件。他让 Claude 创建了一个小脚本，分别在新移植版本和原 Python 代码库上运行 7 个真实场景，并比较结果差异。每个失败场景都分配一个独立修复 Agent，循环持续运行，直到 7 个场景全部通过。

随后他又向前推进了一步：Claude 自行设计端到端测试套件，连续 4 个夜晚自主运行，修复遇到的问题后再次执行。这样发现了任何预设场景清单都很难预料到的细小体验问题。

这里的经验是：缺少测试套件并不会阻止这一阶段。如果不能继承一个现成裁判，就让 Claude 创建一个。无论如何，原始代码库都是行为真值。

## 代码迁移最佳实践

每次运行都会揭示上一轮没有发现的问题。可以肯定，你的下一次迁移也会遇到本指南无法预见的情况。不过，以下实践在所有项目中都经受住了检验：

- **不要盲目照搬本指南。** 每次迁移都不一样。应把本文作为起点，在正式投入前先与 Claude 一起规划适合当前项目的迁移方案。
- **不要把注意力放在单个失败上。** 单个失败属于循环本身的工作，修复 Agent 会逐项消化它们。人类应把注意力放在反复出现的模式上。
- **让审查具有对抗性，让验证保持机械化。** 对抗式审查能够支持更长时间运行的任务，通常值得额外消耗 Token。应让脚本——编译器、差异比较工具和测试套件——充当裁判。
- **不要所有工作都使用最大的模型。** Token 消耗主要集中在循环中，因此要有意识地设计模型分工。较小模型适合大规模并行实施；最大的模型应留给审查工作，以及所有负责制定其他 Agent 所遵循规则的任务。
- **把人类工时前置。** 规则手册和压力测试最耗费人力。完成这些后，后续工作大多只是不断消化任务队列。
- **让工作队列机械化并可恢复。** “完成”的定义应当是：“输出文件已经存在于磁盘上。”

## 审查循环结果，而不是逐行审查代码

Jarred 的 Bun 迁移版本现在已经进入生产环境，不过所有迁移都存在取舍。例如，约 4% 的 Rust 代码位于 `unsafe` 块中，其中大部分是与 C/C++ 边界交互的单行指针操作。

但新代码库已经在可测量的指标上变得更好。团队工具能够检测到的所有内存泄漏都已修复：在一项连续执行 2,000 次构建的基准测试中，内存占用从 6,745 MB 降至 609 MB。Linux 和 Windows 上的二进制文件体积缩小了 19%。跨语言优化还让 HTTP 服务以及 `next build`、`tsc` 等真实工作负载获得了 2%～5% 的性能提升。

也许现在到了重新计算那些长期搁置迁移项目成本收益的时候。选一个你一直勉强忍受的代码库，问问 Claude：它的迁移流程应该是什么样子？
